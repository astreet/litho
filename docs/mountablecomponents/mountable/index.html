Lifecycle of a Mountable

A Mountable represents a reusable unit responsible for hosting the logic to create, measure, and bind the content that the mountable component will render.

As illustrated in the Creating a Mountable Component page, the MountableComponent.render() method should return a Mountable implementation and any Style object to be applied to the component on the MountableRenderResult object.

note

Mountable extends RenderUnit, which is a rendering primitive. For more information, see the Introduction to RenderCore page.

Lifecycle of a Mountableâ€‹

A Mountable has four important stages in its lifecycle, which must occur in the following order:

  1. Content size measurement
  2. Creation
  3. Content creation
  4. Binding and unbinding content properties

Each of these stages is detailed in the following sub-sections.

Content size measurementâ€‹

note

This stage of the mountable's lifecycle can occur on any thread.

Each Mountable should implement a measure() function to define how it measures itself given arbitrary width and height specs. The MeasureResult object it returns contains the width and height of the content, and optionally any layout data, as shown in the following example:

override fun measure(
    context: LayoutContext<*>,
    widthSpec: Int,
    heightSpec: Int,
    previousLayoutData: Any?
): MeasureResult {
  return if (SizeSpec.getMode(widthSpec) == SizeSpec.UNSPECIFIED &&
      SizeSpec.getMode(heightSpec) == SizeSpec.UNSPECIFIED) {
    MeasureResult(defaultSize, defaultSize)
  } else {
    MeasureResult.withEqualDimensions(widthSpec, heightSpec, null)
  }
}

To learn about the different strategies to measure content, see the Measuring page.

Creation of a Mountableâ€‹

In order to create a Mountable, write a class that extends the Mountable abstract class. Implementations directly extending the Mountable class enable multiple Binders to be specified to set and unset properties on the mounted content (see Binding and unbinding content properties).

In the majority of cases, a single Binder will be enough to implement the binding required for the MountableComponent.

For this reason, a SimpleMountable abstract class has been created, which encapsulates the logic of the Binder creation and addition. To implement setting and unsetting the properties of the content, just override mount(), unmount(), and, optionally, the shouldUpdate() method.

The following example provides an implementation of SimpleMountable in an ImageView:

internal class SimpleImageViewMountable() : SimpleMountable<ImageView>(RenderType.VIEW) {

  override fun createContent(context: Context): ImageView = ImageView(context)

  override fun measure(
      context: LayoutContext<*>,
      widthSpec: Int,
      heightSpec: Int,
      previousLayoutData: Any?
  ): MeasureResult {
    return if (SizeSpec.getMode(widthSpec) == SizeSpec.UNSPECIFIED &&
        SizeSpec.getMode(heightSpec) == SizeSpec.UNSPECIFIED) {
      MeasureResult(defaultSize, defaultSize)
    } else {
      MeasureResult.withEqualDimensions(widthSpec, heightSpec, null)
    }
  }

  override fun mount(c: Context, content: ImageView, layoutData: Any?) {
    content.setImageDrawable(c.resources.getDrawable(R.drawable.ic_launcher))
  }

  override fun unmount(c: Context, content: ImageView, layoutData: Any?) {
    content.setImageResource(0)
  }
}

Content creationâ€‹

note

This stage of the mountable's lifecycle can only occur on the main thread.

Each Mountable needs to create the content it hosts (either a View or a Drawable) via a ContentAllocator.

override fun createContent(context: Context): ImageView = ImageView(context)

You should also pass the type of the created content to the Mountable super constructor, either RenderType.VIEW for views or RenderType.DRAWABLE for drawables.

note

The content should not be mutated based on props passed from MountableComponent to the Mountable.

In order to optimize the mount performance, the ContentAllocator can also be used to define content pooling.

Binding and unbinding content propertiesâ€‹

note

This stage of the mountable's lifecycle can only occur on the main thread.

A Mountable uses one or more Binders to bind() and unbind() properties on the mounted content.

Each Binder should implement 3 methods:

  • bind() - used to set properties.
  • unbind() - used to unset properties.
  • shouldUpdate() - used to decide whether properties need resetting.

The following code shows a component that extends SimpleMountable, where mount() is the equivalent of bind() and unmount() is the equivalent of unbind():

override fun mount(c: Context, content: ImageView, layoutData: Any?) {
  content.setImageDrawable(c.getResources().getDrawable(R.drawable.ic_launcher))
}

override fun unmount(c: Context, content: ImageView, layoutData: Any?) {
  content.setImageResource(0)
  content.rotation = 0f
  content.scaleX = 1f
  content.scaleY = 1f
  content.setBackgroundColor(Color.BLACK)
}

Each Binder should be added to the Mountable using the addMountUnmountExtension() method.

Important

Once set, a property should be unset in the matching unbind() method to ensure correctness when the content is reused.
